## Ownership and scientific importance

Issue #5439 is already explicitly owned. Reporter `chummels` is a repository
member, is assigned to the issue, and opened PR #5440 one minute later. The PR is
open, non-draft, mergeable, and targeted at milestone 4.4.3. The author later
asked for review because the fix is needed by downstream Trident work.

The defect is scientifically meaningful and source-established. A FIRE/GIZMO
particle filter evaluates the derived `age` field over data chunks. A chunk with
no `PartType4` star particles produces an empty `StellarFormationTime` array.
GIZMO's `creation_time` field converts it to redshift and unconditionally calls
`Cosmology.t_from_z`; `t_from_a` then calls `min()` on the empty array and raises
the issue-reported reduction error. Valid chunked analysis and derived quantities
therefore fail rather than silently returning the empty particle selection.

This is not an implementation opportunity for a second contributor. The useful
work is to assess the existing patch and avoid duplicating it.

## Patch assessment

The production change is narrow and mechanically sound:

- reads `StellarFormationTime` once;
- returns an empty yt array with the dataset's configured time unit when the
  input has zero elements;
- preserves the existing cosmological conversion for nonempty scale factors;
- preserves the non-cosmological `code_time` conversion for nonempty values.

The empty return agrees with the registered field units and with normal particle
filter behavior: a zero-particle chunk should propagate an empty field and mask.
No numerical clipping, altered nonempty calculation, or cross-frontend refactor is
introduced.

Changing `_creation_time(data)` to `_creation_time(field, data)` is runtime
compatible with current `DerivedField._func_signature_type`, which supports both
signatures. It is nevertheless contrary to an explicit unresolved reviewer
request to keep the existing one-argument signature.

## Findings

### BLOCKING

1. **An explicit reviewer request remains unresolved.**

   Reviewer `jzuhone` asked the author to retain `_creation_time(data)` and
   pinged again on 2026-06-02. The current head still uses
   `_creation_time(field, data)`, and the review thread remains unresolved.

   The two-argument form is not a demonstrated runtime bug, but the requested
   change is unnecessary to the fix and remains a visible merge blocker. Restore
   the one-argument signature before promotion.

### IMPORTANT

1. **The new regression is external-data-dependent and ordinary CI does not
   prove it ran.**

   `test_gizmo_creation_time_handles_empty_particle_chunks` is decorated with:

   ```python
   @requires_file("FIRE_M12i_ref11/snapshot_600.hdf5")
   ```

   `yt.testing.requires_file` skips the test when the file is absent and strict
   requirements are disabled. More decisively, `pyproject.toml` explicitly
   ignores `yt/frontends/gizmo/tests/test_outputs.py` in ordinary pytest jobs.
   The nose jobs have no FIRE-data download or `test_data_dir` configuration, so
   they skip the decorated test on GitHub-hosted runners. The configured GIZMO
   answer suite in `tests/tests.yaml` lists only `test_gizmo_64`, not the new
   regression. The separate combined status `Full testsuite (nose)` remains
   pending and may represent an external data-backed run, but the green ordinary
   Build and Test workflow does not execute this regression.

   The FIRE workflow is a useful integration test when the dataset exists, but
   it must not be the only regression protecting a seven-line empty-array guard.
   Add a normally collected deterministic test that exercises the cosmological
   empty-input path without sample data.

2. **The reviewed branch is stale relative to current `main`.**

   The head was last committed on 2026-04-27 and is 113 commits behind current
   `main`. The visible Actions runs succeeded on that old head, while the
   full-suite status is still pending. Rebase or update the branch and rerun the
   relevant checks before promotion.

3. **The test assertion is too indirect for the new invariant.**

   `assert com.shape == (3,)` proves only that the reported end-to-end workflow
   completed when the FIRE dataset is available. It does not assert that an empty
   `creation_time` result has zero length and the configured time dimensions.
   A focused unit assertion would catch wrong units or a nonempty sentinel even
   if a higher-level derived quantity later tolerated it. That assertion belongs
   in the required deterministic regression.

### OPTIONAL

1. One implementation option for the required regression is a normally collected
   file such as `yt/frontends/gizmo/tests/test_fields.py`. An in-memory
   `load_uniform_grid(..., nprocs=8)` fixture can place one `PartType4` particle
   in one grid and deterministically leave seven grids empty, following the
   repository's stream-particle test pattern. Because `load_uniform_grid`
   defaults to a non-cosmological dataset, the test must explicitly set
   `ds.cosmological_simulation = True` and inject a suitable cosmology spy/stub.
   Register or capture GIZMO's `creation_time` function, then assert the fixture
   precondition and result:

   - per-grid particle counts include both zero and one;
   - each result size equals that grid's formation-time input size;
   - every result has time dimensions;
   - `cosmology.t_from_z` is called for the one nonempty grid only and never for
     an empty input.

   If GIZMO registration cannot be applied directly to the stream field info, a
   recorder object can invoke
   `GizmoFieldInfo.setup_star_particle_fields` and retain the function passed to
   `add_field`. Its fake data/DS must likewise set
   `cosmological_simulation=True` and make `t_from_z` fail if called for the empty
   array. Either approach avoids a GIZMO file, HDF5, particle index, and archive
   download. Keep the FIRE workflow as optional integration coverage.

2. A separate audit may check other frontend particle fields that call
   `t_from_z` on possibly empty arrays. Do not broaden this PR without a distinct
   reproduced failure.
