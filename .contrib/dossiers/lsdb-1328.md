## Decision boundary

LSDB's non-fine spatial search is deliberately coarse for the main catalog: it
keeps every row in each HEALPix partition overlapping the requested region and
does not run the point-level geometric filter.

An attached margin follows the same search object, but HATS widens margin
partition selection to neighboring pixels so boundary-near duplicates are not
lost. With `fine=False`, LSDB also skips point filtering for those widened
margin partitions. Complete margin partitions can therefore survive even when
some of their rows lie outside both the exact requested region and the coarse
main-catalog pixels.

This is the source-confirmed behavior. The issue additionally reports rows up
to about one HEALPix-pixel width from the requested region and "strange"
crossmatches; neither distance nor final output was measured locally.

Current downstream source concatenates right-margin rows with the right main
partition before crossmatch and join algorithms run. That proves the extra rows
are available to downstream operations. It does **not** by itself prove a false
spatial match: a crossmatch algorithm may reject a row by radius, and a keyed
join changes only if alignment and key values permit it.

Decision: **GO for this source-only dossier; WAIT for runtime work,
implementation, and upstream contact.** BuildKit PR #6966 remains the
portfolio's sole active implementation.

## Scientific and operational importance

HATS margins duplicate sources near HEALPix boundaries into neighboring
partitions. LSDB consumes those duplicates to keep partition-local
crossmatches and joins complete without loading a whole survey catalog.

Non-fine search is a performance feature for science workflows that need a
rough sky cut. The current interaction can make its right-side candidate data
broader than the coarse main result. Source-supported risks include:

- additional rows, memory, and distance or key comparisons in each affected
  task;
- distant margin duplicates becoming eligible for a keyed join when their key
  collides with a left row;
- output membership or duplicate behavior depending on partition order,
  margin threshold, and search geometry rather than only the requested region;
- confusing coverage and provenance because the main and margin structures
  describe different widened footprints.

The following are not established:

- a false result from LSDB's default distance-limited crossmatch;
- an incorrect result from a real catalog join;
- an affected public dataset, scientific analysis, or publication;
- the frequency of non-fine searches followed by margin-consuming operations;
- the runtime or memory magnitude at survey scale.

The scientific concern is therefore a credible reproducibility and
performance boundary, not evidence of a known incorrect published result.

## Source mechanism

### One search object reaches both catalogs

[`Catalog.search()`](https://github.com/astronomy-commons/lsdb/blob/00396644a31655b49bfe44270f7ede5c62b47fdb/src/lsdb/catalog/catalog.py#L790-L808)
first searches the main catalog and then calls `self.margin.search(search)` with
the same `AbstractSearch` instance.

The region search classes in
[`region_search.py`](https://github.com/astronomy-commons/lsdb/blob/00396644a31655b49bfe44270f7ede5c62b47fdb/src/lsdb/core/search/region_search.py)
carry the `fine` flag and provide two separate operations:

1. filter the HATS structure to overlapping partitions;
2. filter rows by their coordinates.

Cone, box, polygon, and MOC searches share this split. `PixelSearch` is a
MOC-based search whose default is already non-fine.

### Fine controls row filtering

The base
[`HealpixDataset.search()`](https://github.com/astronomy-commons/lsdb/blob/00396644a31655b49bfe44270f7ede5c62b47fdb/src/lsdb/catalog/dataset/healpix_dataset.py#L1218-L1259)
always filters the HATS structure and selects its partitions. It wraps those
partitions in `MapPartitions(search.search_points, ...)` only when
`search.fine` is true.

For a coarse search,
[`AbstractSearch.filter_hc_catalog()`](https://github.com/astronomy-commons/lsdb/blob/00396644a31655b49bfe44270f7ede5c62b47fdb/src/lsdb/core/search/abstract_search.py#L31-L55)
also replaces the filtered structure's exact MOC with the MOC of its selected
pixel tree. The main result consequently represents full selected pixels, as
the coarse API intends.

### Margin partition selection is deliberately wider

The HATS
[`MarginCatalog.filter_by_moc()`](https://github.com/astronomy-commons/hats/blob/1dff6b2ca67d476534fb9cfae94dd1943194e15e/src/hats/catalog/margin_cache/margin_catalog.py#L13-L43)
documents that margin filtering must include every possibly overlapping
partition and may include extras. It:

1. degrades the input MOC when the configured margin threshold exceeds the
   resolution of the MOC's highest order;
2. adds neighboring HEALPix cells;
3. filters the margin pixel tree with that expanded MOC.

Because the margin's HATS structure is a `MarginCatalog`, this override runs
during the same region search. When `fine=False`, the subsequent LSDB base
search does not call `search_points` on the selected margin frames. Rows from
the neighbor-expanded partitions remain unfiltered.

### Structural result

| Layer | `fine=True` | `fine=False` |
| --- | --- | --- |
| Main partitions | Overlap the exact request | Overlap the exact request |
| Main rows | Geometrically filtered | Every row in selected partitions |
| Main MOC | Exact filtered coverage | Selected-pixel MOC |
| Margin partitions | Neighbor-expanded superset | Neighbor-expanded superset |
| Margin rows | Geometrically filtered | Every row in selected margin partitions |
| Margin MOC | Original margin coverage intersected with the neighbor-expanded filter MOC | Neighbor-expanded selected-pixel MOC |

Fine row containment therefore does not imply an exact structural margin MOC.
For a coarse search, the main result is wider than the exact region and the
margin can be wider again because it starts from neighbor-expanded partition
selection.

## Downstream propagation

### Crossmatch

[`crossmatch_catalog_data()`](https://github.com/astronomy-commons/lsdb/blob/00396644a31655b49bfe44270f7ede5c62b47fdb/src/lsdb/operations/functions/crossmatch_catalog_data.py#L298-L408)
aligns the left and right catalogs with `add_right_margin=True`, passes
`right.margin` into the partition operation, and
[`perform_crossmatch()`](https://github.com/astronomy-commons/lsdb/blob/00396644a31655b49bfe44270f7ede5c62b47fdb/src/lsdb/operations/functions/crossmatch_catalog_data.py#L25-L180)
concatenates the right partition and its margin before invoking the selected
algorithm. The nested crossmatch path does the same.

Verified conclusion: an extra retained margin row enters the right-side input
frame.

Unverified conclusion: whether it becomes an output row. That depends on
alignment, the chosen crossmatch algorithm, its radius and duplicate rules, and
the synthetic geometry.

### Joins

[`perform_join_on()` and `perform_join_through()`](https://github.com/astronomy-commons/lsdb/blob/00396644a31655b49bfe44270f7ede5c62b47fdb/src/lsdb/operations/functions/join_catalog_data.py#L30-L285)
concatenate the right partition with its margin before key matching.
`perform_join_nested()` does the same when an aligned right pixel exists; when
`right_pixel` is absent it uses `right_meta` instead. Their dispatch paths pass
`right.margin` for aligned right pixels.

Verified conclusion: a retained margin row with an eligible alignment and
matching key can participate in a join.

Unverified conclusion: that ordinary data contains such a collision or that a
specific public result changes. `merge_asof` does not consume margins and is
outside this issue's path.

## Contract options from the issue

The issue proposes three directions. None is an accepted maintainer contract.

| Proposal | Completeness | Candidate/output risk | Laziness and cost | API surprise |
| --- | --- | --- | --- | --- |
| Ignore margin pixels outside the requested region | The issue says boundary matches may be missed; the exact completeness domain is undefined | Fewer extra rows, but no proof that every inappropriate candidate disappears | Could preserve partition-only execution if implemented structurally | A catalog still has a margin, but it may no longer protect every coarse main row |
| Drop the margin | Loses the most boundary support; later operations already warn when a right margin is absent | Removes margin-derived candidates entirely | Lowest row-filter cost and memory | High unless the API clearly signals or warns about the loss |
| Fine-filter the margin while leaving the main coarse | Reporter-preferred and intended to retain useful matches, but "all" depends on whether completeness covers the exact region or every coarse main row | Removes geometrically distant margin rows | Can remain lazy, but adds point-level work to margin partitions at execution and weakens the partition-only performance expectation | Mixed semantics: main is coarse while margin is fine |

The third proposal has a specific unresolved edge: a coarse main search keeps
rows outside the exact geometric request but inside selected pixels. Filtering
the margin to the exact request may remove boundary support for those
coarse-only rows. Filtering instead to the main result's selected-pixel MOC is
a different contract. Source inspection cannot choose between them.

## Unexecuted tiny-fixture protocol

Start with the repository-local `small_sky_order1_source_with_margin` fixture.
Do not download a survey catalog.

Construct or identify three distinguishable source rows:

1. **exact-region row:** geometrically inside the request;
2. **coarse-only row:** outside the exact request but inside a selected main
   pixel;
3. **neighbor-margin row:** retained through the neighbor-expanded margin
   partition and outside the intended comparison footprint.

Record pixels, MOCs, row IDs, coordinates, and margin provenance separately for
the main and margin catalogs.

| Case | Structural assertion | Row assertion | Downstream control |
| --- | --- | --- | --- |
| Fine cone | Main and margin select required partitions | Main and margin rows lie inside the cone | Legitimate boundary match remains |
| Coarse cone | Main selects overlapping pixels; margin records its expanded pixels | Distinguish exact, coarse-only, and neighbor-margin rows | Small radius rejects the distant row; record candidate input separately |
| Fine/coarse MOC or box | Repeat with a second geometry | Confirm the behavior is not cone-specific | Same boundary and distant controls |
| Keyed join | Use deliberate distinct and colliding keys | Track which right rows enter the merge | Show whether the distant row changes membership under each key |
| Empty region | No main partitions | No margin partitions or rows | Empty result |
| Catalog without margin | Main behavior unchanged | No margin frame | Existing warning/completeness behavior remains explicit |
| Large threshold | MOC degradation and neighbor selection are recorded | Separate partition coverage from row containment | No claim about correctness from structure alone |

The crossmatch control needs both:

- a legitimate source across a HEALPix boundary and within the configured
  radius, so a candidate fix cannot silently reduce completeness;
- a retained distant source outside the radius, so candidate-frame inclusion
  is not mislabeled as a false match.

The keyed-join control needs deliberate key collision and non-collision cases.
Expected final membership must not be encoded until maintainers choose the
completeness contract.

Suggested future focused command, only after authorization and a checkout:

```text
python -m pytest tests/lsdb/catalog/test_cone_search.py -q
```

The final accepted regression may belong beside join or crossmatch tests rather
than only in `test_cone_search.py`.

## Maintainer decisions required

1. **Completeness domain:** every row returned by the coarse main catalog, or
   only rows inside the exact geometric request.
2. **Coarse safety:** whether a coarse-filtered catalog must remain safe for
   later margin-consuming joins and crossmatches without a warning.
3. **Margin footprint:** exact request, coarse main pixel MOC, or
   neighbor-expanded margin MOC.
4. **Behavior:** narrow the margin, drop it, fine-filter it, or expose a new
   explicit policy.
5. **Search scope:** cone, box, polygon, MOC, pixel, loading-time search
   filters, and any non-spatial searches that attach a margin.
6. **Metadata:** what pixel tree, MOC, and margin-presence signal must accompany
   a narrowed or dropped margin.
7. **Laziness and performance:** acceptable row-level work and memory overhead
   for coarse searches on survey-scale catalogs.
8. **Compatibility:** warning, behavioral fix, opt-in mode, and release-note
   treatment.

Do not select among these choices without maintainer direction.
