## Why this matters

Ray Serve's OpenAI-compatible LLM ingress includes `str(e)` in the response sent
to an API caller. For `RayTaskError`, that string contains the remote traceback,
including internal paths, node IPs, process and actor identifiers, stack frames,
and potentially secrets embedded in exception messages.

The same exception is already passed to the server-side logging handler with
`exc_info=e` (repeated fatal-engine logs may be rate-limited), so returning it
to the client adds no diagnostic capability for operators. It expands an
ordinary failed inference request into an information-disclosure path.

## Current-source evidence

`python/ray/llm/_internal/serve/utils/server_utils.py` still contains:

```python
if status_code == status.HTTP_500_INTERNAL_SERVER_ERROR:
    internal_message = message = "Internal Server Error"
    exc_type = "InternalServerError"
else:
    internal_message, message = _extract_message(e)
    exc_type = e.__class__.__name__

error_info = ErrorInfo(
    message=(
        f"Message: {message}, Internal exception: {internal_message}, "
        f"original exception: {str(e)}"
    ),
    code=status_code,
    type=exc_type,
)
```

The reported HTTP 500 path therefore restores the original exception immediately
after replacing the public message with `Internal Server Error`.

## Scope is broader than exact 500 responses

Sanitization uses equality with `500`, not the 5xx class. An exception carrying
`status_code=501`, `502`, `503`, or another server-error code follows
`_extract_message(e)` and then appends `str(e)`. Those paths can expose the
exception both as the selected message and as the explicitly appended original
exception.

The security invariant should be:

> No 5xx response contains exception text, traceback text, internal class paths,
> or other server-only diagnostic material.

For 4xx responses, the intended public validation or request message may remain,
but the response should not append the internal exception representation.

## Existing test gap

`python/ray/llm/tests/serve/utils/test_error_log_ratelimiter.py` has one
`TestGetResponseForError` test. It verifies that:

- the server-side fatal error handler is called;
- an error response is returned;
- the request ID appears in the response.

It does not assert that the response excludes the exception or traceback, and it
does not cover non-500 5xx status codes or preservation of safe 4xx messages.

## Minimal test-first patch boundary

Add focused unit tests beside `TestGetResponseForError` before changing
production code.

### 1. Exact 500 must not disclose

Use an exception whose text contains a unique sentinel such as
`secret-connection-string`. Assert:

- response code is `500`;
- type remains `InternalServerError`;
- message contains `Internal Server Error` and the request ID;
- message does not contain the sentinel;
- message does not contain `original exception` or `Internal exception`.

### 2. Other 5xx responses must not disclose

Use a small exception class with `status_code = 503` and a secret-bearing
message. Assert the same non-disclosure invariant, preserve code `503`, and
normalize the public type to `InternalServerError`. This prevents an
exact-500-only fix and makes the intended 5xx type behavior explicit.

### 3. Expected 4xx messages remain useful

Use `OpenAIHTTPException(status_code=400, message="invalid model")`. Assert:

- code remains `400`;
- the public message and request ID remain;
- no internal exception representation or fully qualified class path appears.

### 4. Logging remains diagnostic

Patch `_fatal_error_log_handler.log` and assert the original exception is still
passed to it. Client sanitization must not reduce server-side observability.

## Smallest production change

Keep status selection, logging, and request-ID handling in place. Change only the
public message construction:

1. Treat every `status_code >= 500` as a server error for message
   sanitization.
2. For 5xx, expose only `Internal Server Error` plus the request ID.
3. For 4xx, expose only the selected public `message` plus the request ID.
4. Never append `internal_message` or `str(e)` to `ErrorInfo.message`.

Do not combine this security fix with the adjacent OpenAI envelope-compliance
work tracked in #64833. Fields such as `type`, `param`, and `code` can be
corrected independently; the non-disclosure patch should remain auditable and
easy to backport.

## Compatibility considerations

- Some clients or tests may parse the current `Message: ..., Internal
  exception: ...` string despite it not being a safe public contract. Search
  before editing and treat removal as an intentional security hardening.
- Preserve the request ID so operators can correlate a sanitized client failure
  with the full server log.
- Do not merely strip ANSI codes or traceback-looking lines. Exception messages
  themselves can contain secrets, and `RayTaskError` formatting is not the only
  disclosure source.
- Do not add a configuration switch that re-enables raw exceptions in public
  responses.
