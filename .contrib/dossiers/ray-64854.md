## Outcome

The report accurately describes Ray 2.55.1: that release obtains
`DeltaTable.file_uris()` and passes the physical Parquet files to
`read_parquet()`. This bypasses Delta's logical-to-physical column mapping and
can expose UUID-style physical names.

That code path no longer exists on Ray master. Merged PR
[#61721](https://github.com/ray-project/ray/pull/61721) changed
`read_delta()` to use `DeltaTable.to_pyarrow_dataset()` and
`ParquetDatasource.from_pyarrow_dataset()`.

With the report's exact deltalake and PyArrow versions, the current path does
not silently return a wrong schema. It rejects the column-mapped table with a
`DeltaProtocolError` because deltalake 1.6.1 does not support reader protocol
version 2.

This is safer than the release behavior, but it is not column-mapping support.

## Standalone reproduction

The reusable fixture is under `reproducers/ray-64854/`. It requires no
Databricks account, cloud storage, or Ray cluster.

It creates a minimal protocol-gate/mechanism fixture:

- a Parquet file containing physical column
  `col-4d014c4b-7d15-4d7b-a89e-40dfdca76a22`;
- a Delta transaction log declaring name-based column mapping;
- a logical Delta field named `customer_score`.

Executed with:

```text
Python 3.11
deltalake 1.6.1
pyarrow 21.0.0
```

Command:

```sh
python reproducers/ray-64854/reproduce.py
```

Observed:

```text
raw file schema: ['col-4d014c4b-7d15-4d7b-a89e-40dfdca76a22']
delta schema: ['customer_score']
unified dataset error: The table's minimum reader version is 2 but deltalake only supports version 1 or 3 with these reader features: {'variantType-preview', 'variantType', 'timestampNtz'}
PASS: the old raw-file path exposes the physical name, while the current unified path fails explicitly
```

The ordering of the reported feature set is nondeterministic; the assertions
check only the stable protocol-error clauses.

### Fixture limitation

The Parquet file is written with plain PyArrow and does not encode Delta's
protocol-required column `field_id`. It is not a fully conforming
column-mapped writer output.

That limitation does not invalidate the two narrow conclusions:

- name-mode readers resolve data columns by
  `delta.columnMapping.physicalName`, not by field ID, so the raw schema
  demonstrates the physical-name exposure mechanism;
- deltalake 1.6.1 rejects reader protocol v2 before it scans the Parquet file.

## Version-specific source evidence

### Ray 2.55.1

```python
paths = DeltaTable(path, version=version).file_uris()
return read_parquet(paths, ...)
```

This constructs a generic Parquet dataset from raw files. The reproducer proves
that its schema uses the physical UUID name.

### Current master

```python
dt = DeltaTable(path, version=version, storage_options=storage_options)
pa_dataset = dt.to_pyarrow_dataset(filesystem=filesystem)
datasource = ParquetDatasource.from_pyarrow_dataset(pa_dataset, ...)
```

The Delta reader now controls schema and protocol handling before Ray constructs
its datasource. With deltalake 1.6.1, the unsupported table fails before any
Ray scan, projection, cast, or row filter can run.

## Root-cause boundary

Delta's protocol requires a reader using name-based column mapping to resolve
physical Parquet names through each field's
`delta.columnMapping.physicalName` metadata. Applying a Ray-side rename after
raw file discovery is not a complete implementation:

- nested fields also require mapping;
- partition values and statistics use physical names;
- files missing mapped columns must produce nulls;
- predicate pushdown must bind against physical fields while the public schema
  remains logical;
- protocol feature/version checks must happen before reading.

Those semantics belong in the Delta reader. Current Ray correctly delegates to
delta-rs rather than reimplementing part of the protocol.

## Row-filter claim

The report also says a logical-name row filter can be silently ignored in the
Unity Catalog workaround. This fixture does not exercise Unity Catalog or a
filter, so that claim remains unverified and must not be inferred from the
schema result. It is not the behavior of current master with deltalake 1.6.1:
construction raises `DeltaProtocolError` before a filter reaches the Ray
datasource.

Any future column-mapping implementation must include a filter test to ensure
logical predicates are translated and pushed down correctly; schema-only
coverage is insufficient.

## Recommended upstream disposition

Do not create a Ray patch that manually renames UUID columns.

Recommended issue response:

1. Confirm that the silent wrong-schema behavior is specific to Ray 2.55.1's
   pre-#61721 implementation.
2. Ask the reporter to test a current nightly/master build.
3. Explain that current master fails explicitly with deltalake 1.6.1 rather
   than returning physical columns.
4. Track full column-mapping support against delta-rs (or whichever Delta
   reader Ray adopts), including logical predicate translation.
5. If desired, open a separate Ray usability issue for a clearer
   column-mapping-specific error message.

## Promotion boundary

This is not a Ray implementation candidate in its current form. It becomes one
only if:

- a supported Delta reader exposes a correct mapped PyArrow dataset and Ray
  corrupts it afterward; or
- maintainers explicitly choose to add a clear, test-backed compatibility error
  around the upstream `DeltaProtocolError`.

Until then, the useful contribution is the local reproducer and an evidence-led
triage comment, not a partial rename patch.
