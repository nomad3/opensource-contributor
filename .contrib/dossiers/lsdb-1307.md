## Decision boundary

LSDB exposes user-defined transforms that preserve a catalog's HATS spatial
structure while accepting arbitrary output frames. The public API says that
right ascension and declination are assumed not to change, but current source
does not compare their input and output values.

That creates a source-confirmed invariant gap:

1. a UDF can return changed RA or Dec values;
2. the row's `_healpix_29` key, output partition, pixel tree, MOC, and margin
   membership can still describe the original sky position;
3. no coordinate-consistency error is guaranteed before another spatial
   operation consumes the result.

The issue itself classifies in-place coordinate changes as illegal and names
`map_partitions`, `map_rows`, and `merge_map` as safeguard targets. Its only
comment clarifies that `meta` cannot establish value preservation: true
validation has to compare computed input and output coordinates.

The defect is important, but the implementation contract is not ready. In
particular, `_healpix_29` is not guaranteed to be unique, and partition UDFs
may filter, reorder, or duplicate rows. Maintainers must define how an output
row is matched to its input before a value comparison can be correct and
compatible.

Decision: **GO for this source-only dossier; WAIT for runtime work,
implementation, and upstream contact.** BuildKit PR #6966 remains the
portfolio's sole active implementation.

## Scientific and operational importance

LSDB targets spatial analysis of survey-scale astronomical catalogs. HATS
partitions the sky into HEALPix pixels so Dask can prune work and perform
spatial operations without loading an entire catalog.

The pinned
[HATS guide](https://github.com/astronomy-commons/hats/blob/1dff6b2ca67d476534fb9cfae94dd1943194e15e/HATS_GUIDE.md)
establishes that:

- `_healpix_29` is a fixed-order spatial key derived from a row's sky
  coordinates;
- coarser partition membership is recovered from that key;
- the key supports row-level spatial filtering;
- a pixel tree and MOC describe the partitions and sky coverage;
- a margin catalog duplicates boundary-near objects into neighboring pixels so
  edge-spanning crossmatches and joins remain complete.

The current LSDB margin generator likewise computes membership from coordinates
with `radec2pix` in
[`margin_catalog_generator.py`](https://github.com/astronomy-commons/lsdb/blob/00396644a31655b49bfe44270f7ede5c62b47fdb/src/lsdb/loaders/dataframe/margin_catalog_generator.py#L185-L251).

If only RA or Dec changes, those representations can disagree. Supported
risks include:

- a search near the new coordinate pruning the old partition before seeing the
  moved row;
- a crossmatch aligning work by the old pixel and failing to compare the row
  with candidates near its new position;
- stale margin membership omitting a newly boundary-near object or retaining
  an inappropriate duplicate;
- writing the transformed result as a durable, internally inconsistent HATS
  catalog.

These are source-supported consequences of the broken invariant, not locally
reproduced incidents. No affected public catalog, missed association, wrong
published result, or incidence rate was established.

The issue's motivating Gaia epoch-propagation workflow avoided the unsafe path
by writing corrected positions to new `ra_corr` and `dec_corr` columns. This
dossier did not retrieve or execute the linked Rubin notebook and does not
claim that it produced an invalid catalog.

## Spatial invariant map

| Representation | Normal source of truth | What remains after a coordinate-changing UDF |
| --- | --- | --- |
| Declared RA/Dec columns | HATS `TableProperties` names plus row values | Names can remain valid while values change |
| `_healpix_29` | Computed from RA/Dec | Existing index is retained unless the UDF explicitly changes it |
| Partition | Coarser HEALPix membership derived from the spatial key | UDF output remains in the task's original or aligned output partition |
| Pixel tree | Structural list of catalog partitions | Reused or derived from input alignment, not rebuilt from UDF coordinates |
| MOC | Structural sky coverage | Reused from the input structure or alignment |
| Margin membership | Boundary proximity calculated from coordinates | The same UDF may alter copied rows, but the duplicate set is not regenerated |

Changing `_healpix_29` together with RA/Dec is not a safe workaround. The row
would still need to move to the correct partition, and pixel-tree, MOC, and
margin structure might need rebuilding. The issue asks for prevention, not a
partial reindexing path.

## Entry-point matrix

### `map_partitions`

The base implementation is in
[`healpix_dataset.py`](https://github.com/astronomy-commons/lsdb/blob/00396644a31655b49bfe44270f7ede5c62b47fdb/src/lsdb/catalog/dataset/healpix_dataset.py#L302-L427).
Its docstring states that each row's RA and Dec are assumed unchanged. The
ordinary path wraps the UDF in `MapPartitions`; the updated dataset refreshes
schema and row-count metadata but retains the catalog's structural pixels and
MOC.

The `Catalog` wrapper in
[`catalog.py`](https://github.com/astronomy-commons/lsdb/blob/00396644a31655b49bfe44270f7ede5c62b47fdb/src/lsdb/catalog/catalog.py#L811-L940)
applies the same partition UDF to an attached margin. This updates values in
the existing main and margin rows; it does not regenerate which rows belong in
the margin.

`compute_single_partition=True` materializes one selected partition and
constructs the result with that same pixel. It has the same invariant risk but
a different, eager error-timing boundary.

### `map_rows`

The base row transform is in
[`healpix_dataset.py`](https://github.com/astronomy-commons/lsdb/blob/00396644a31655b49bfe44270f7ede5c62b47fdb/src/lsdb/catalog/dataset/healpix_dataset.py#L1581-L1745).

Current behavior has two materially different modes:

| Mode | Current safeguard | Remaining boundary |
| --- | --- | --- |
| `append_columns=True` | Rejects output names that overlap existing columns, including RA/Dec | A UDF cannot replace coordinate columns through this mode |
| replacement mode | Retains declared coordinate metadata when the returned `meta` includes the existing names | Returned RA/Dec values are not compared with the input row |

The existing focused test returns unchanged RA and Dec in replacement mode. It
checks successful metadata propagation, not rejection of changed values.

The `Catalog` wrapper in
[`catalog.py`](https://github.com/astronomy-commons/lsdb/blob/00396644a31655b49bfe44270f7ede5c62b47fdb/src/lsdb/catalog/catalog.py#L1301-L1437)
also applies `map_rows` to an attached margin without rebuilding membership.

### `merge_map`

[`Catalog.merge_map()`](https://github.com/astronomy-commons/lsdb/blob/00396644a31655b49bfe44270f7ede5c62b47fdb/src/lsdb/catalog/catalog.py#L635-L692)
accepts a UDF for each aligned point-catalog and map-catalog partition. The
execution path in
[`merge_map_catalog_data.py`](https://github.com/astronomy-commons/lsdb/blob/00396644a31655b49bfe44270f7ede5c62b47fdb/src/lsdb/operations/functions/merge_map_catalog_data.py#L25-L160)
sorts input partitions, optionally filters by the original spatial index, then
returns the UDF result.

The output catalog uses the point catalog's metadata with the alignment's pixel
tree and MOC. It does not compare output coordinates or rebuild spatial
structure from them. The existing test returns its point-catalog partition
unchanged and therefore does not exercise the unsafe case.

## Evidence and inference boundary

Verified:

- HATS derives the spatial key and margin membership from coordinates.
- the three named UDF surfaces preserve structural metadata without comparing
  coordinate values;
- replacement-mode `map_rows` can return the existing coordinate names with
  different values;
- `map_partitions` and `map_rows` apply the UDF separately to an existing
  margin rather than regenerate margin membership;
- #1345 rejected metadata-only validation as incomplete and disruptive;
- no public direct implementation was found at the audit timestamp.

Not established:

- a runtime reproduction on current LSDB;
- the exact output of a cone search or crossmatch after a boundary-crossing
  mutation;
- a real catalog or scientific analysis affected by this behavior;
- the frequency of coordinate-changing UDFs;
- the accepted row-correspondence, equality, error, or reimport contract;
- the runtime and memory cost of a correct guard at survey scale.

## Unexecuted synthetic boundary protocol

Use only existing tiny repository fixtures. Do not download Rubin, Gaia, or
other survey data.

1. Select a row in a small multi-partition catalog.
2. Derive a coordinate immediately across an adjacent HEALPix boundary with
   the pinned HATS helpers.
3. Return that coordinate from each target UDF while deliberately retaining the
   original `_healpix_29`.
4. Record the output coordinate, retained index, actual Dask partition,
   structural pixel tree and MOC, and attached margin membership.
5. Compare fine cone-search and crossmatch behavior at the original and moved
   positions.
6. Keep every predicted miss or stale-margin effect labeled unverified until
   the test executes.

Smallest future contract matrix:

| Path | Passing control | Rejection case |
| --- | --- | --- |
| `map_partitions` | unchanged RA/Dec plus a derived column; empty partition | RA-only and Dec-only mutation |
| `map_rows` | replacement returning unchanged RA/Dec | RA-only and Dec-only mutation |
| `merge_map` | current point catalog plus `tests/data/square_map` | RA-only and Dec-only mutation |
| Attached margin | unchanged transform on main and margin | mutation detected in either execution path |
| Laziness | graph construction performs no full-catalog compute | ordinary error occurs on execution; single-partition mode errors during its eager call |
| Row identity | filtered, reordered, and duplicate-index controls | a changed coordinate cannot hide behind another row with the same index |

A within-the-same-partition coordinate shift is also required. It proves that a
future guard compares values rather than checking only whether the output row
still belongs to the original coarse pixel.

No test in this matrix was written or executed.

## Maintainer decisions required

1. **Row correspondence:** how to match surviving output rows to inputs after
   filtering, reordering, or duplication when `_healpix_29` may be non-unique.
2. **Comparison semantics:** exact numerical equality versus pandas-aware
   handling of NaN, signed zero, dtype conversion, and coordinate units.
3. **Scope:** all dataframe-returning `map_partitions` paths, replacement-mode
   `map_rows`, `merge_map`, margins, and single-partition execution.
4. **Missing or renamed coordinates:** immediate rejection, a deliberately
   non-spatial result, or treatment only under #1457.
5. **Timing:** preserve lazy Dask execution without an eager full-catalog scan;
   give partition and method context when a worker detects a mutation.
6. **Intentional motion:** hard error plus documented reimport/repartition
   workflow, or a separate explicit API designed to rebuild every HATS
   invariant.
7. **Compatibility and performance:** acceptable per-partition memory and CPU
   overhead for catalogs approaching billions of rows.

Do not select among these choices without maintainer direction.

compute, shuffle, network transfer, or repartition. Its expected algorithmic
cost is at least linear in output rows and may require snapshots of two input
columns because a UDF can mutate its input frame in place. That cost is an
estimate, not a benchmark result. A future accepted design needs a pass-case
ASV benchmark.
