## Scientific and operational impact

Dask is core execution infrastructure for chunked xarray and array pipelines
used across astronomy, Earth observation, climate, and other data-intensive
research. Issue #12507 reports that deterministic task keys can optimize into
different graph shapes solely because Python starts with another string hash
seed.

The supplied toy graph varies by roughly 2.5× in the report. The reporter's real
xarray pipeline varies between 127,166 and 4,048,385 optimized tasks and between
61 and 1,010 seconds of graph-construction time. In bad draws, slicing is not
pushed through rechunk and overlap layers, so a small spatial selection retains
the global read and reprojection pipeline.

This is a reproducibility and scalability defect. The inspected evidence does
not show an incorrect numerical result or silent scientific-data corruption;
the established harm is nondeterministic graph size, startup time, scheduler
load, and avoidable upstream I/O work.

[`reproducers/dask-12507`](../../reproducers/dask-12507). It uses the public issue
graph unchanged in the behaviorally relevant places:

- 1,600 deterministically named delayed read tasks;
- concatenation into an 8×10×20 block layout;
- `map_overlap` with eight-pixel spatial depth;
- rechunking to `(8, 512, 512)`;

## First-divergence localization

The reproducer follows the phase order in
[`dask/array/optimization.py::optimize`](https://github.com/dask/dask/blob/a54329156cd17a68ef081c83fdc2409ddb423426/dask/array/optimization.py):

| Phase | Seed 0 | Seed 1 | Seed 7 |
| --- | ---: | ---: | ---: |
| Raw high-level graph | 3,302 layers / 27,593 tasks | same | same |
| After `optimize_blockwise` | 3,301 / 25,993 | same | same |
| After `fuse_roots` | 3,301 / 25,993 | same | same |
| After `HighLevelGraph.cull` | 762 / 1,206 | 454 / 746 | 358 / 612 |
| After low-level linear fusion | 728 tasks | 428 | 334 |

Canonicalized layer-name and declared layer-dependency fingerprints are
identical across the three seeds through `fuse_roots`; task and layer counts
first differ after culling. This is not a complete structural fingerprint:
adversarial inspection found 1,600 fused `_trim` tasks whose internal argument
or placeholder ordering already differs across seeds after blockwise
optimization. Slice fusion does not change the already-divergent low-level task
counts.

Current
[`HighLevelGraph._toposort_layers`](https://github.com/dask/dask/blob/a54329156cd17a68ef081c83fdc2409ddb423426/dask/highlevelgraph.py)
builds reverse-dependency lists by iterating dependency sets, appends newly
ready layers to a list, and pops from that list. Hash order can therefore change
which valid topological order is returned. `HighLevelGraph.cull()` consumes the
reverse of that order while mutating a shared required-key set across layers.

As a causal experiment, the reproducer replaces only `_toposort_layers()` with
a deterministic min-heap traversal during culling. All eight seeds then
produce:

```text
after cull: 465 layers / 776 tasks
after low-level fusion: 438 tasks
```

This isolates hash-sensitive layer traversal as the trigger for the first
task-count and layer-topology divergence in this graph. It does **not** make
sorting the fix: the 438-task graph is not minimal, and seed-specific task
representations remain. Crossed-order controls show that a chosen traversal
order, rather than the seed that built the pre-cull graph, determines the
over-retained task count.

## Root-cause candidate and fixed-path control

Merged [PR #11899](https://github.com/dask/dask/pull/11899), commit
[`ef1c013`](https://github.com/dask/dask/commit/ef1c0131b62f5150a9d31e689990927b97d222a1),
changed `HighLevelGraph.cull()` in May 2025. The previous code intersected the
global outstanding-key set with each layer's output keys before calling
`layer.cull()`. The new code passes the full global set to every layer.

That interacts unsafely with two cardinality fast paths introduced or retained
by the same change:

- legacy `Layer.cull()` treats `len(keys) == len(layer)` as a request for the
  entire layer;
- `_task_spec.cull()` returns the full graph when the requested-key count equals
  the graph count.

Those checks assume every supplied key belongs to the layer. A foreign key set
with the same cardinality can therefore retain an unrelated whole layer.
Hash-sensitive traversal changes which outstanding foreign keys reach a layer
and whether the false fast path fires.

The reproducer's `--cardinality-control` constructs two-key legacy and task-spec
layers, supplies two entirely foreign keys, and confirms that both cull paths
retain every owned key. This direct control isolates the unsafe assumption
without relying on the larger array graph. It does not yet instrument which
shortcut fires for each seed in the end-to-end array graph.

The local control restores only per-layer ownership filtering:

```python
output_keys = keys_set.intersection(layer.get_output_keys())
culled_layer, culled_deps = layer.cull(output_keys, all_external_keys)
```

Without sorting topological traversal, every seed from 0 through 7 then produces
exactly 82 optimized tasks. Each synthetic source block encodes its time,
row-block, and column-block coordinates. Seed 0 and seed 7 controls both compute
a `float64` array with shape `(8, 100, 100)` whose time planes contain the
expected values `0, 10000, ..., 70000`; a wrong input block changes that oracle.
This is substantially stronger than the 438-task sorting experiment and is the
current narrow root-cause candidate, not an upstream patch.

## Overlap and design history

No pull request, branch, or commit matching #12507, `PYTHONHASHSEED`, or
high-level-graph culling determinism was found. Active
[PR #12470](https://github.com/dask/dask/pull/12470) refactors the same
pre-cull blockwise/root-fusion calls and extends them to delayed and bag graphs.
It does not claim #12507 or change the array cull call, but its head must be
included in the pre-implementation recheck.

Three older pull requests touch adjacent optimizer structure but do not claim
this issue:

- [#8560](https://github.com/dask/dask/pull/8560) and
  [#9273](https://github.com/dask/dask/pull/9273) explore moving culling before
  `fuse_roots`; both remain drafts;
- [#10534](https://github.com/dask/dask/pull/10534) explores lazy overlap-layer
  materialization; it is open, non-draft, conflicting, and dormant.

They are not active ownership, but their design direction must be reviewed
before choosing a patch.

Merged [PR #7403](https://github.com/dask/dask/pull/7403) introduced the current
linear-time layer topological sort as a performance rewrite without a
determinism regression. Issue
[#8535](https://github.com/dask/dask/issues/8535) and merged
[PR #8542](https://github.com/dask/dask/pull/8542) are useful precedent that
different blockwise traversal orders can expose correctness problems, but they
are not duplicates of #12507.

## Smallest future regression

The strongest upstream tests should prove both key ownership and end-to-end
seed independence:

1. construct a materialized or task-spec layer with `N` keys;
2. pass `N` foreign keys and assert culling does not take the “all keys
   requested” fast path;
3. build a reduced overlap → rechunk → slice graph with deterministic keys;
4. run it in subprocesses with known divergent hash seeds;
5. assert the same required graph keys and task count;
6. assert the optimized collection computes the same selected array;
7. retain an optimizer-performance guard because #11899 was a speedup.

The same-cardinality unit regression is the smallest causal test. The
subprocess graph regression is slower but preserves the public failure and
guards interactions among blockwise fusion, layer ordering, and culling.

## Narrow implementation boundary

The first candidate boundary is `HighLevelGraph.cull()` in
`dask/highlevelgraph.py`, not xarray, delayed key generation, numerical array
code, or `_toposort_layers()`. Restore per-layer output-key filtering, or make
the lower-level cardinality fast paths validate key ownership before returning
the full layer. Preserve #11899's performance objective and test both legacy
and task-spec layers.

Do not sort every optimizer dependency or propose deterministic topological
ordering as the fix. Blockwise task representations can differ before culling,
but the per-layer ownership control makes the final graph invariant under the
native seed-dependent orders.
